## 1. 何を比較するかの問題提起

Claude API で広告文のバリエーションを作らせるとき、プロンプトの組み方次第で出力の多様性・トーンの一貫性・コストが大きく変わる。「一発で複数パターンを出させる」「構造化出力で強制的に型を揃える」「生成→自己評価→改善のループを回す」の3つの設計を、同じ入力条件で比較して何が違うのかを検証する。

## 2. 比較対象の整理

3つのプロンプト設計パターンを整理する:

- パターンA: 単発プロンプトで複数バリエーションを一括生成
- パターンB: tool use / JSON Schema で構造化出力を強制し、型を固定して生成
- パターンC: 生成 → 自己評価(基準に沿っているか自己採点) → 改善、を1〜2周回すループ

各パターンの狙いと想定されるトレードオフ(応答速度・トークンコスト・出力の安定性)を軽く整理する。

## 3. 検証手順(コード)

同一の入力条件(商品カテゴリ・トーン指定・文字数上限などの制約)を3パターンそれぞれに与えて実行するスクリプトを組む。評価観点は以下を予定:

- 出力のバリエーション幅(似たような文言に収束していないか)
- 制約(文字数・禁止表現・トーン)の遵守率
- 1回の生成にかかるトークン数 / レイテンシ

## 4. 結果と考察

3パターンの実行結果を並べ、どの観点でどれが優れていたか・どこに弱点が出たかを考察する。「型を固定すると多様性が落ちる」「ループを回すとコストが跳ねる」といった、実際に手を動かして初めて分かるトレードオフを具体的に書く。

## 5. データアナリスト視点

評価観点の設計(遵守率・バリエーション幅をどう定量化するか)を、データ分析での指標設計の視点と絡めて2〜3文で触れる。
