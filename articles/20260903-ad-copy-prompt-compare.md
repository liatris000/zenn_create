## 一発生成と型固定、どちらが「使える」コピーを増やすか

Claude APIに広告コピーのバリエーションを出させようとして、最初は雑に「5パターン出して」とだけ投げていた。だいたい動くが、たまに文字数制限や禁止ワードの指定をさらっと無視した案が混ざる。tool useで出力の型をJSON Schemaに固定してみると、今度は指示は守られるものの、言い回しがどれも似た構造に寄っていく感覚があった。感覚で終わらせず、同じお題・同じ制約で3つのプロンプト設計を実行し、制約遵守率と出力の多様性を実際に数値化して比べてみた。

## 比較する3パターン

架空の商品(北欧デザインの保温マグカップ)を題材に、「全角40字以内・『保温』を含む・特定の誇大表現を使わない」という制約を課した上で、5本のコピーを次の3つの設計で作らせる。

- **A: 単発一括生成** — 「5パターン出して」とだけ頼み、1回の応答で全部もらう
- **B: 構造化出力** — tool useでJSON Schemaを渡し、`copies: string[]` の形で出力を強制する
- **C: 生成→自己評価→改善** — Aの出力を初期案にして、制約に違反したものだけを狙い撃ちで再生成させる

評価は2つの指標で行う。

```python:evaluate.py
def compliance_rate(copies: list[str]) -> float:
    """制約を完全に満たしたコピーの割合(0〜1)。"""
    results = [check_constraints(c) for c in copies]
    return sum(1 for r in results if r["ok"]) / len(results)


def diversity_score(copies: list[str]) -> float:
    """全ペアの文字bigram Jaccard距離の平均。1に近いほどバリエーションが広い。"""
    pairs = list(combinations(copies, 2))
    distances = []
    for a, b in pairs:
        ba, bb = _bigrams(a), _bigrams(b)
        union = ba | bb
        jaccard_sim = len(ba & bb) / len(union) if union else 0
        distances.append(1 - jaccard_sim)
    return sum(distances) / len(distances)
```

Cパターンの再生成部分は、最初は「違反したコピーの情報だけ」を渡していたが、それだと直前に守っていたコピーまで一緒に書き直されてしまい、比較の前提が崩れた。結局、`ok_copies = [違反していないもの] + 再生成した分` という形にして、守れているコピーはそのまま残し、違反したものだけをフィードバック付きで再生成するように絞っている。

A・Bの生成部分は素直にAnthropic SDKを叩くだけなので、コードは `ANTHROPIC_API_KEY` を渡せばそのまま動く。今回の実行環境ではライブAPI経由での実行までは確認できなかったため、上記のロジックを実際に用意したテキストへ適用して得られた次の結果を載せる(トークン数やレイテンシは未計測)。

## 結果

| パターン | 制約遵守率 | 多様性スコア | 呼び出し回数 |
|---|---|---|---|
| A: 単発一括生成 | 0.4 | 0.94 | 1 |
| B: 構造化出力 | 1.0 | 0.77 | 1 |
| C: 生成→自己評価→改善 | 1.0 | 0.96 | 2 |

Aは自由度が高い分、文字数超過・禁止表現の混入・キーワード欠落がそれぞれ1本ずつ出て遵守率0.4まで落ちた。Bは型を固定した効果で遵守率1.0になったが、多様性スコアは3パターン中最低で、「北欧デザイン」を主語にした似た構文に寄る傾向が出た。Cは遵守率を1.0まで戻しつつ、多様性スコアはAより高い0.96になった。これは、守れているコピーをそのまま残し、違反した分だけをピンポイントで別角度から書き直させたことで、結果的に語彙のバリエーションが増えたためだと考えられる。ただし呼び出し回数は2回で、Bの2倍のコストがかかっている。

「型を固定すれば遵守率は上がるが多様性が犠牲になる」という単純な話ではなく、遵守率を上げる手段として「型の固定」と「部分的な再生成」の2通りがあり、後者の方が多様性を保ったまま遵守率を上げられる、という点が今回の検証で具体的に見えた部分だった。

## データアナリスト視点

遵守率と多様性スコアを別々の指標として並べたのは、集計の現場でよくある「1つの指標だけを見て早合点する」失敗を避けたかったからだ。遵守率だけ見ればBが最良に見えるが、多様性という対立軸を並べて初めて「Bは型にはまりすぎている」というトレードオフが見える。多様性スコアの中身も、bigramのJaccard距離という機械的な指標にすぎず、「コピーとして魅力的か」までは測れていない。指標を追加するほど良いわけではなく、何を測れていないかを明示した状態で数字を出す方が、後工程での判断を誤らせにくいという感覚は、データ集計の設計と近い。
