## 問い合わせの受け口からして、すでに手作業だった

問い合わせが届くたびに、読んで・転記して・確認して・テンプレを探して……という流れがどのチームでも固定化している。ツールを入れるたびに「どこに情報が落ちているか」が増えていく。

Claude API を使ってこのフロー全体をパイプライン化できないか検証した。分類・追加情報収集・契約書生成を1フローに収めるのがゴール。

---

## 設計の全体像

<!-- Day 2 で Mermaid フロー図を追記 -->
パイプラインは以下の3フェーズで構成する:

1. **分類フェーズ** — 受信した問い合わせを Claude で種別・優先度・業種にタグ付け
2. **収集フェーズ** — 情報が不足している場合、Claude が質問テンプレートを生成して担当者へ渡す
3. **生成フェーズ** — 必要情報が揃ったら Claude が契約書ドラフトを生成

各フェーズの状態は SQLite で管理し、フェーズ遷移をログとして残す。この設計にした理由は「自動化しすぎない」ことにある。フェーズ 2 の収集は人間が送るので、そこで意図的に止まる構造にした。

---

## Claude API の `tool_use` をどう使うか

<!-- Day 2 でツール定義コードを追記 -->
3フェーズそれぞれに tool を定義する。単純なプロンプトではなく tool_use にした理由は、分類結果を構造化した形で受け取りたいから。JSON で返させようとするより信頼性が高い。

- `classify_inquiry` — 種別・優先度・業種をタグとして返す
- `generate_followup_questions` — 不足情報の質問を箇条書きで返す
- `draft_contract` — 情報セットを受け取り契約書本文を返す

最初はプロンプトで JSON を返させていたが、フォーマット崩れが頻発したので tool_use に切り替えた。

---

## 「いつ止まるか」の設計が本質

全自動にしたい誘惑がある。けれど問い合わせ対応の品質は「止まるタイミング」で決まる。

- 分類結果が信頼できない場合の fallback をどう設計するか
- 収集フェーズで担当者が追加情報を返さなかったらどうするか
- 契約書ドラフトが確認なしに送られる状態にしない

この3点を設計段階で決めておかないと、運用に乗せた後に「AI が勝手に動いた」トラブルになる。承認が必要なステップはどこか、というのを先に決める。

---

## データアナリスト視点から見たこの設計

問い合わせの各フェーズをDB に記録する設計は、後から分析できる基盤になる。「どの種別の問い合わせが成約につながりやすいか」「収集フェーズで詰まる問い合わせパターンはどれか」を後から SQL で取り出せる。

ログを捨てずにためておくだけで、運用データが分析資産になる。問い合わせ管理と分析基盤は同じ DB 設計で両立できる。

## この設計を使ってみて

問い合わせ管理の文脈では「自動化の手前の設計」が一番むずかしい。Claude がどこまでやって、どこで人間に返すか。この分界点を先に決めてから実装するほうが、後から設計を壊す確率が下がる。

コードが動く前から「止まるポイント」を図に書いておく習慣が、AI パイプライン実装には思ったより重要だった。
