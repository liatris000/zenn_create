海外クラウドファンディングで製品を仕入れる際、候補リストの精査に毎回似たような判断をしている。「日本でこれが売れるか」「競合はどのくらいいるか」「輸送コストは見合うか」——この評価軸は変わらないのに、なぜか毎回手で考えている。Claude API の構造化出力なら、この判断を Pipeline として実装できると気づいた。

## スコアリングの設計思想

製品評価に使う軸を整理すると、大きく4つに分解できる。各軸に 1〜5 のスコアを Claude に付けてもらい、合計で採用判定を出す設計にした。

- **市場適合性**: 日本市場での需要見込み（類似商品の普及状況）
- **競合密度**: Amazon.co.jp 等での既存商品の飽和具合
- **製品新規性**: カテゴリ内でのユニークな差別化要素の有無
- **事業性**: 想定コストに対しての利益余地

各評価軸を別々に API を叩くのではなく、1 回の呼び出しで全軸をまとめて採点させる方針にした理由と、構造化出力の設計で悩んだポイントを後述する。

## アーキテクチャ概要

```
製品情報(テキスト) → Claude API (structured output) → スコアJSON → 判定ロジック → CSV出力
```

入力は製品タイトル・説明文・価格帯・カテゴリの4フィールド。Claude API の `tools` パラメータを使って、スコアの型と値域を強制する。出力は JSON で受け取り、CSV に積み上げる。

### Step 1: 評価スキーマの定義

Claude に渡す tool スキーマで、各軸のスコア型（integer, 1-5）とその評価基準（description）を定義する。ここが一番設計コストのかかる箇所で、description の書き方で出力の質が大きく変わる。

### Step 2: バッチ呼び出しの実装

候補製品リスト（CSV）をループして Claude API に逐次投げる処理。レートリミットを避けるためのウェイトの挿入位置や、リトライ設計のポイントを書く。

### Step 3: スコア集計と判定

4 軸の合計スコアに閾値を設けて「要調査 / 見送り / 有力」の 3 段階に分類する。閾値の決め方は後述。

## 試行錯誤したところ

最初は評価軸ごとに別の API 呼び出しにしていたが、1 製品あたり 4 回コールする実装は遅くコストも高かった。また、各軸が独立して評価されるため「市場適合性は高いが競合が多い」という矛盾スコアが出やすかった。

1 回の呼び出しにまとめることで、Claude が軸間の整合性を保ちながら採点するようになり、矛盾が減った。structured output の schema に reasoning フィールドを追加して、なぜそのスコアを付けたかを一言残させるのも効いた。

## データアナリスト視点

製品評価スコアを CSV に蓄積すれば、軸ごとの分布や「採用率との相関が高い軸はどれか」が見えてくる。売上データとスコアを突合すれば、スコアモデルの精度を測るフィードバックループが作れる。この構造は SQL での集計前に特徴量エンジニアリングを設計する作業と近い——どの軸をどう定義するかで結果が変わる。
