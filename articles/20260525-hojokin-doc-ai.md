補助金申請の書類作成は、構造が決まっているのに毎回ゼロから書いている。事業計画書の「現状と課題」「実施内容」「期待効果」は、入力する情報が変わるだけで骨格は同じだ。Claude API でそのパターンをそのまま使えないか試したら、思ったより実用的なものができた。

## 補助金申請書類の構造を理解する

<!-- Day 2 で書く内容:
  - 代表的な補助金(ものづくり補助金, IT導入補助金等)の書類構造の共通パターン
  - セクション一覧: 現状と課題 / 事業計画 / 実施体制 / 期待効果 / 費用計画
  - なぜ「構造が決まっている」と言えるのかを示す
-->

中小企業が申請する代表的な補助金(ものづくり補助金・IT導入補助金等)の申請書類には共通のセクション構造がある。そこを自動化対象として切り出す。

## アーキテクチャ

<!-- Day 2 で書く内容:
  - 入力: 会社概要 YAML + 申請補助金の種類 + 事業概要(自由テキスト)
  - Claude API: セクション単位で分割生成
  - 出力: Markdown 形式の申請書類ドラフト
  - Mermaid フロー図も入れる
-->

入力は「会社情報」「事業内容」「申請する補助金の種類」の3つ。Claude API にセクション単位で生成させ、Markdown で出力する。一発生成ではなくセクション分割にする理由は Day 2 で実装しながら説明する(先に試して失敗した)。

## 実装: 入力形式の設計

<!-- Day 2 で書く内容:
  - company_profile.yaml の形式(会社名NG、業種・規模・課題のみ)
  - 補助金タイプの enum 定義
  - 入力バリデーション
  - コード: input_schema.py

会社の固有名詞ではなく「業種」「従業員規模」「直面している課題の種別」を入力として受け取る設計にする。個人情報を Claude に渡さなくて済む構造を先に決める。

## 実装: プロンプト設計

補助金申請書類には「審査官が読む」という特性がある。プロンプトに審査基準の観点を明示的に持たせると、生成文の質が変わる。どう持たせるかは実装時に確認する。
