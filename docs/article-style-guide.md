# 記事スタイルガイド

zenn.dev/liatris に公開する記事の文体・構成の方針を定義する。

**最終更新**: 2026-04-30
**対象**: Claude Code が生成する記事 + Liatris が編集する記事
**目的**: AI 記事のテンプレ感を脱却し、Liatris らしさが出る記事にする

## このガイドの位置づけ

`zenn_create` の Day 1〜3 で生成される記事は、本ガイドに従って書かれる。
本ガイドは [`.claude/skills/article-writing/SKILL.md`](../.claude/skills/article-writing/SKILL.md) から自動参照される。

## 何を最適化するか

優先順位順:

1. **転職市場露出** ─ データアナリストとしてのキャリア軸が見える
2. **キャッチアップ価値の自己開示** ─ Liatris が何を学んで何を考えたかが分かる
3. **読者への貢献** ─ 動くコード + 業務応用イメージ
4. **AI らしさからの距離** ─ テンプレ的構成・定型表現を避ける

## 何を最適化しないか

- 短期的な PV / バズ
- SEO のための機械的な見出し量産
- 全方位網羅(範囲を絞る方が深く書ける)
- 「綺麗にまとまっている」こと(綺麗さ ≒ AI 臭)

## やらないこと(禁止リスト)

### 表現レベルの禁止

#### リード文のテンプレ表現

- ❌「〜と感じたことはないでしょうか」
- ❌「〜で悩んでいる方も多いのではないでしょうか」
- ❌「結論から言うと、〜」
- ❌「今回は〜について解説します」

リード文は **題材固有の問題提起** から書き始める。

例(良い): 「Claude Code の Hooks を業務スクリプトに組み込んでみたら、想定外に dbt 開発フローが楽になった。」
例(悪い): 「Claude Code Hooks を使って開発フローを自動化したいと感じたことはないでしょうか。今回はその方法を解説します。」

#### 構成のテンプレ

- ❌「良かった点 / 惜しかった点 / 業務での活用イメージ」を毎記事に置かない
- ❌「まとめ」セクションを毎記事に置かない(自然に終われるなら不要)
- ❌「こんな方に試してほしいです」のリストで記事を締めない

これらは「AI が記事を書いた」と即座に判別される構造。題材ごとに構成を変える。

#### 文末のテンプレ

- ❌「いかがでしたか」
- ❌「ぜひ試してみてください」
- ❌「参考になれば幸いです」

これらは何も伝えていない決まり文句。題材固有の余韻を残す。

### 内容レベルの禁止

#### 業務コンテクストの露出

- ❌ 業務文脈・関係者・取引先・パートナー・提携先を出さない
- ❌「家業で困っていた」「DLG で作った」「kubell で議論されている」等の動機を書かない
- 詳細は `business-profile/policies/disclosure-rules.md` 参照

#### 本名の露出

- ❌ 記事本文内に「平野翔斗」と書かない
- ✅ プロフィール表示は本名 OK

#### フィクションの数値

- ❌「クエリ実行時間が 1.5 秒から 0.3 秒に改善」のような未確認の数値を書かない
- ✅ 実際に計測した結果のみ書く

## やること(必須要素)

### 必須セクション(順序自由)

各記事に以下を入れる。順序や重み付けは題材に応じて変える。

#### A. データアナリスト視点(必須)

**目的**: キャリア軸が記事の中で見えるようにする。

「この実装をデータ分析業務に応用するなら」「データアナリストとしてこの題材を見ると」
といった視点を、記事のどこかに 2〜3 文程度で入れる。

例:

- 「OCR 結果を構造化データに落とし込む発想は、SQL の集計前処理と同じ構造をしている。」
- 「Claude Code Hooks の PreToolUse で .env をブロックする仕組みは、データ基盤側の row-level security と相似だ。」

ただし以下は避ける:

- ❌「データアナリストとして〜」を強調する露骨な自己紹介
- ❌ kubell や Voicy 等の固有業務コンテクストを出すこと

#### B. ハマった点 / 思考プロセス(必須)

**目的**: AI 記事の「綺麗にまとまっている」を脱却する。

実装中に試行錯誤した点、最初に試した方法と最終的な方法の違い、なぜそれを選んだか
を 1〜2 か所に入れる。「失敗談」「ハマった点」と書くのではなく、自然に文章に織り込む。

例(自然):

> 最初は `PreToolUse` の matcher を `*` で全ツール対象にしていたが、SessionStart まで
> 巻き込まれてログが汚れたので `Edit|Write|Bash` に絞った。

例(不自然):

> 【ハマった点】matcher の指定で困りました。最初は `*` でしたが、これは全ツールが対象に
> なるため不適切でした。

#### C. 動くコード(必須)

**目的**: 読者が手を動かせる。

- コードは省略せず、コピペで動く完全な形で書く
- 言語指定を必ずつける(```bash:scripts/hooks/session-start.sh のように)
- 環境依存(Mac / Linux / Windows)があれば明記

### オプション要素(題材に応じて入れる)

#### 既存ツールとの比較

類似機能を持つ他ツールがある場合、簡潔に位置づけを書く。
過剰に他ツールを下げない、上げない。

#### 図(Mermaid)

処理フローやアーキテクチャを示す時は Mermaid で図示すると効果的。
ただし図のための図は不要。文章で済むなら文章で。

#### Pros / Cons の見せ方

「良かった点」「惜しかった点」のような対比は、**やる場合は具体性で勝負する**。

- ❌「セットアップが簡単で良かった」
- ✅「`.claude/settings.json` 1 ファイルで動く。npm パッケージ等の追加依存ゼロ」

## 構成パターン(参考)

題材によって以下のパターンを使い分ける。毎記事で違うパターンを使う。

### パターン 1: 実装編

1. 題材固有のリード(なぜこれを作ったか)
2. ざっくりとしたアーキテクチャ
3. 実装ステップ(コード中心)
4. 思考プロセス(自然に織り込む)
5. データアナリスト視点(必須)
6. 成果物リンク + スクショ

### パターン 2: 比較・検証編

1. 何を比較するかの問題提起
2. 比較対象の整理
3. 検証手順(コード)
4. 結果と考察
5. データアナリスト視点(分析的に何が言えるか)

### パターン 3: 概念解説編

1. 概念の抽象的なイントロ
2. なぜそれが重要か
3. 具体例(コード)
4. 自分なりの解釈
5. データアナリスト視点

(これら以外でも、題材に合えば自由に構成して良い)

## 文字数の目安

- 1500〜3000 字を基本とする
- 短すぎる(< 1000 字)→ 内容が薄い可能性、追記検討
- 長すぎる(> 4000 字)→ 焦点がぼけている可能性、分割または削減

## チェックリスト(記事完成時)

Day 3 のセルフレビューで使う:

- [ ] リード文がテンプレ的でない(冒頭の禁止表現を確認)
- [ ] 「結論から言うと」「いかがでしたか」等の定型表現がない
- [ ] データアナリスト視点が 2〜3 文入っている
- [ ] ハマった点 / 思考プロセスが自然に織り込まれている
- [ ] 業務コンテクスト(家業・DLG・kubell 内部等)が出ていない
- [ ] 本名「平野翔斗」が記事本文内にない
- [ ] フィクションの数値がない
- [ ] コードが動く形で書かれている(言語指定あり)
- [ ] 文字数が 1500〜3000 字の範囲
